# Дизайн: тематизация через CSS-переменные + слоты

**Дата:** 2026-05-21
**Статус:** утверждён к реализации
**Базируется на:** библиотека `bp-mui-date-range-picker`

## Цель

Сейчас стили библиотеки завязаны на Tailwind: вид по умолчанию — Tailwind-классы
на элементах, потребитель обязан иметь Tailwind. Нужно, чтобы внешний вид можно
было менять и **без Tailwind** — через CSS-переменные, при этом существующий
слот-механизм (`classNames`) сохраняется. Два слоя сосуществуют с фиксированным
приоритетом: слот всегда побеждает тему.

Это изменение уровня фундамента — переписывается весь слой стилей.

## 1. Архитектура — два слоя

**Слой 1 — тема (переменные).** Библиотека поставляет рукописный `styles.css`.
У каждой части компонента — стабильный класс с префиксом `drp-` (`drp-input`,
`drp-day`, `drp-day--selected` и т.д.). `styles.css` задаёт вид этих классов;
каждое значимое свойство берётся из CSS-переменной
(`background: var(--drp-accent, <запасное>)`). Все селекторы обёрнуты в
`:where(...)` → **нулевая специфичность**.

**Слой 2 — слоты (`classNames`).** Проп `classNames` навешивает классы
потребителя поверх стабильного `drp-`-класса. Базовый CSS имеет нулевую
специфичность, поэтому любой класс потребителя его перебивает — приоритет
«слот выигрывает» получается автоматически, без зависимости от порядка
импортов. Конфигурируемый приоритет сознательно НЕ делается (CSS-приоритет
плохо переключается в рантайме; фиксированный «слот выигрывает» — то, что нужно
почти всегда).

Использование потребителем:
```ts
import 'bp-mui-date-range-picker/styles.css';        // базовый вид
```
```css
.my-app { --drp-accent: #16a34a; --drp-radius: 10px; } /* тема — без Tailwind */
```
```tsx
<DateRangeInput classNames={{ daySelected: 'bg-emerald-600' }} /> /* точечный override */
```

Каждый элемент всегда получает свой `drp-`-класс; классы из `classNames` — сверху.
Tailwind для потребителя становится **необязательным** — нужен лишь если он сам
пишет слот-override Tailwind-классами.

## 2. Набор CSS-переменных (~28 токенов)

Значения по умолчанию определяются на `.drp-root` и воспроизводят текущий вид
(палитра zinc/indigo).

**Цвета (~16):** `--drp-accent`, `--drp-accent-fg`, `--drp-bg`, `--drp-fg`,
`--drp-muted-fg`, `--drp-border`, `--drp-hover-bg`, `--drp-range-bg`,
`--drp-range-fg`, `--drp-today-fg`, `--drp-disabled-fg`,
`--drp-shortcut-active-bg`, `--drp-shortcut-active-fg`, `--drp-invalid-border`,
`--drp-invalid-ring`, `--drp-focus-ring`.

**Форма и размеры (~8):** `--drp-radius`, `--drp-day-size`,
`--drp-input-height`, `--drp-nav-button-size`, `--drp-popover-shadow`,
`--drp-panel-padding`, `--drp-shortcuts-width`, `--drp-gap`.

**Типографика (~4):** `--drp-font-family`, `--drp-font-size`,
`--drp-font-size-sm`, `--drp-font-weight-medium`.

Потребитель переопределяет любой токен — глобально, на обёртке или на инстансе
(`style={{ '--drp-accent': '...' }}`).

## 3. Изменения в коде

**Новый файл `src/styles.css`:**
- `.drp-root { --drp-accent: …; … }` — дефолты всех токенов.
- `:where(.drp-input) { … }`, `:where(.drp-day--selected) { … }` и т.д. —
  правила для каждого `drp-`-класса, значения через `var(--drp-*)`, селекторы в
  `:where()`.
- Сюда же переезжает позиционирование, которое сейчас задаётся Tailwind-классами:
  стрелки `navLayout="around"` (слева/справа), скрытие `caption_label`.

**`src/styles/defaultClassNames.ts`** — словарь `Slot → Tailwind-строка`
становится `Slot → стабильный drp-класс` (`input: 'drp-input'`,
`daySelected: 'drp-day--selected'`, …). Имя `defaultClassNames` сохраняется
(смысл — «базовый класс слота»).

**`src/utils/mergeClassNames.ts`** — `mergeSlot` почти без изменений:
`twMerge(clsx(defaultClassNames[slot], override, ...extra))`. Подмешивает
`drp-`-класс; `tailwind-merge` по-прежнему разрешает конфликты среди
Tailwind-классов потребителя.

**Компоненты** — вызовы `mergeSlot('слот', classNames, …)` и имена слотов те же,
поэтому большинство компонентов не меняется. Исключение — `RangeCalendar`: его
`rdpClassNames` переписывается — Tailwind-«экстры» (`'hidden'`,
`'absolute left-0 top-0'`) убираются, вместо них `drp-`-классы
(`drp-nav-button--prev` / `--next`, класс для скрытого `caption_label`), а
позиционирование/скрытие переезжает в `styles.css`.

**Сборка (`tsup.config.ts`)** — `src/styles.css` копируется в `dist/styles.css`.

**`package.json`** — в `exports` добавляется `"./styles.css": "./dist/styles.css"`.

**`react-day-picker/style.css`** — структурный CSS rdp; импорт в
`RangeCalendar.tsx` остаётся (раскладка сетки). `styles.css` — отдельный
визуальный/темовый слой, его импортирует потребитель.

**Playground** — импортирует `../src/styles.css`; добавляется демо темизации
через `--drp-*`.

**README** — таблица токенов, пример темы, примечание про приоритет слотов.

**`src/types.ts`** — не меняется (`Slot`, `ClassNames` — те же).

## 4. Обработка ошибок

Не требуется — это изменение слоя представления. Отсутствующая переменная
покрывается запасным значением в `var(--drp-*, <fallback>)`.

## 5. Тестирование

Ограничение: jsdom не применяет внешние стили и не вычисляет CSS (ни каскад, ни
`var()`). Поэтому «переменная поменяла цвет» и «слот перебил тему» проверяются
визуально в playground, не в юнит-тестах.

Автоматически:
- **`mergeClassNames.test.ts`** — переписывается под новую модель:
  `mergeSlot('separator')` → `'drp-separator'`; с override — базовый
  `drp-`-класс + класс потребителя; `tailwind-merge` дедупит конфликтующие
  Tailwind-классы среди override.
- **Компоненты** — каждый элемент несёт свой стабильный `drp-`-класс;
  `classNames`-override добавляется поверх.
- **`styles.css` (проверки содержимого файла)** — файл существует; определяет
  дефолты токенов на `.drp-root`; темовые правила обёрнуты в `:where(`
  (нулевая специфичность — гарантия инварианта «слот выигрывает»).
- **Существующие поведенческие тесты** (календарь, поповер, ввод) остаются
  зелёными.
- **Сборка** — `dist/styles.css` появляется; `exports['./styles.css']`
  резолвится.

Визуальная проверка (playground, вручную): тема через `--drp-*` реально
перекрашивает компонент; слот-override бьёт тему.

## Объём

Одна цельная фича — переработка слоя стилей. Один спек, многозадачный план.
Изменение поведения по умолчанию: для базового вида теперь нужен импорт
`styles.css` вместо настройки Tailwind; Tailwind у потребителя становится
необязательным. Публичный API (`Slot`, `classNames`, пропы) не меняется.
